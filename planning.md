# Planning (Декомпозиция и планирование)

## Описание
Навык для декомпозиции сложных задач. Помогает разложить задачу на подзадачи, составить план и выполнить его. Использует доступные навыки.

## Когда активировать
- Пользователь просит план, стратегию или этапы
- Задача крупная, абстрактная, неоднозначная или рискованная
- Задача затрагивает разные слои проекта

## Принципы
1. **Изучение прежде всего** — сначала понимание, потом планирование.
2. **Навыки и инструменты важны** — адаптируйся.
3. **Определи критерии успеха** — каждая подзадача валидируется.
4. **Установи зависимости** — правильный порядок задач.
5. **Будь конкретным** — чётко формулируй шаги.
6. **Думай о результате** — фокус на целях пользователя.

## Цикл

### Шаг 1: Анализ входных данных
1. Проанализируй исходный запрос и контекст.
2. Сформулируй уточняющие вопросы при необходимости.
3. Изучи проект: файловую структуру, архитектуру, паттерны, зависимости.
4. Изучи документацию к проекту и фреймворкам.
5. Определи доступные навыки и инструменты.
6. Выдели участки проекта, соотносящиеся с требованиями.

### Шаг 2: Планирование
1. Разбей постановку на задачи по структуре проекта.
2. Для каждого шага определи подходящий навык.
3. Пересмотри план, проверь выполнимость.
4. Покажи план пользователю:
```
1. [ ] **<Описание задачи 1>**
    - Навыки: ...
    - Файлы:
        - <путь1>
        - <путь2>
2. [ ] **<Описание задачи 2>**
    - Навыки: ...
    - Файлы:
        - <путь1>
        - <путь2>
```
5. При отрицательном ответе — вернись к анализу и задай уточняющие вопросы.
6. При положительном — запиши план в `.koda/plans/<имя-плана>.md`.

### Шаг 3: Реализация плана
1. Начинай с первого пункта до последнего.
2. Не переходи к следующему, пока текущий не завершён.
3. Шаг завершён когда все запланированные изменения внесены и проверены.
4. Отмечай выполненные шаги: `[ ]` → `[x]`.
5. Если план требует корректировки — возвращайся к анализу.

## Ограничения
- План должен быть наглядным и понятным.
- Если задача простая (до 2 шагов) — план не нужен, переходи к выполнению.
- Не вноси изменения, пока план не подтверждён.
- Не смешивай домены в подзадаче (фронтенд, бэкенд, инфра — отдельно).
- Следуй KISS, YAGNI, DRY, SRP.

## Типичные примеры
- **Веб-приложение:** Backend API → Backend QA → Frontend UI → Frontend QA → Интеграция
- **Микросервисы:** Разработка → Тестирование → Деплой
- **Исправление бага:** Поиск причин → Исправление → Тестирование
- **Рефакторинг:** Анализ → Изменения → Тестирование → Проверка

## Scope
Для простых задач (до 2 шагов) план не нужен. План сохраняется в `.koda/plans/`. Не переходи к следующему шагу, пока текущий не завершён.
